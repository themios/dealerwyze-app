  <name>Task 2: Classify all call sites and write triage document</name>
  <files>.planning/service-role-triage.md</files>
  <action>
Read `.planning/service-role-triage-raw.txt` line by line. For each call site, read the surrounding file context (read the actual file) to determine the classification:

**LEGITIMATE** — Do not change these:
- Any file under `app/api/cron/` — cron runs without user session, service client required
- Storage URL signing (`createSignedUrl`, `getPublicUrl`, `upload`) — RLS cannot sign storage
- Inbound webhook handlers (`twilio/inbound`, `stripe/webhook`, `telegram`, `gmail`) — no user session available
- `app/api/admin/` routes that call `requirePlatformSuperAdmin()` — platform admin, correct
- `lib/sms/` utilities — called outside user session context (outbound SMS)

**REDUCIBLE** — Could use auth client instead:
- Handler calls `requireProfile()` (has a user session) AND uses service client AND all queries include explicit `.eq('org_id', profile.org_id)` or equivalent filter
- The service client is not needed here — RLS on the auth client would enforce the filter automatically
- Record: file path, line number, current pattern, recommended change

**WRONG / NEEDS REVIEW** — Potential security gap:
- Service client used but no explicit org scoping visible in the handler
- Lib utility that creates its own service client without receiving org context as a parameter
- Any pattern where cross-org data could be read without an org filter

**createClientForRequest — CONDITIONAL** (separate category):
- Record all 64 usages separately
- Note: normal sessions → RLS-enforced auth client; staff impersonation sessions → full service-role
- Flag any routes using this that do NOT have explicit org scoping (these are over-privileged during impersonation)
